## Part C: implementation_plan.md のレビュー観点

Claude Code が `implementation_plan.md` を出してきたら、以下の観点で読んでください。**問題があれば追加質問する、なければ実装開始を指示する**、というシンプルな運用です。

### C.1 必ず確認する観点

#### 観点1: 「分かったふり」がないか

Claude Code は時に、仕様の曖昧な部分を勝手に埋めて進めようとする傾向があります。「セクション0: 仕様書からの不明点・追加質問」が空または極端に少ない場合、警戒してください。実際には仕様書に曖昧な点(例: ユーザー数の上限、セッション管理の方針、エラー時の通知優先度等)が複数あるはずです。

**チェック方法**: 「他にも質問はないですか?ここまでの仕様書で本当に全部の判断基準が揃っていますか?」と聞き返す。

#### 観点2: 技術スタックの理由が薄くないか

「FastAPIにしました」だけではなく、「なぜFlaskではなくFastAPIなのか」「なぜDjangoではないのか」が書かれているべきです。仕様書セクション14.1の推奨を踏襲しているだけではNG、その推奨を採用する積極的な理由が要ります。

**チェック方法**: 各選定の「却下した代替案」が書かれているか確認。

#### 観点3: AI Provider抽象化レイヤの拡張性

これが本プロダクトの中核設計です。Phase 1 ではGeminiのみ実装しますが、Phase 2 でClaude/OpenAI/Perplexityを追加するときに、**コード変更が最小** で済む設計になっているかを確認してください。

具体的には:
- 抽象基底クラスのインターフェースが、各Providerの実装可能性をカバーしているか(例: Geminiの safety_settings、Claudeの system prompt、OpenAIのfunction calling等を抽象化できているか)
- ストリーミング応答にも対応できる設計か
- プロンプトテンプレートの変数差し込みが、各Providerの仕様差を吸収できるか

**チェック方法**: 「Claude API を Adapter として後で追加するとき、何箇所のコード変更が必要ですか?」と聞く。

#### 観点4: マルチテナント分離の実装

論理分離(`tenant_id` カラム + アプリ層フィルタ)だけだと、コードバグで他テナントのデータを見せる事故が起きやすい。PostgreSQL の Row Level Security (RLS) を使う方針が望ましい。

**チェック方法**: 「RLS は使いますか?使わない場合、tenant_id フィルタの抜け漏れをどうやって防ぎますか?」と聞く。

#### 観点5: VPS環境調査の具体性

「Nginx 設定を確認しました、衝突ありません」だけでは不十分。実際の `nginx.conf` の関連箇所、稼働中プロセス、空きポート、ディスク・メモリの数値が示されているべき。

**チェック方法**: `docs/vps_environment.md` の中身を見る。曖昧な記述があれば再調査を依頼。

#### 観点6: Phase 1 Week別チケットの実現可能性

各週のタスクが「数だけ書いてある」のではなく、各タスクの工数が現実的か確認。1週間で40時間以上のチケットが詰まっている週は危険信号。バッファ(20〜30%程度)が含まれているかも見る。

**チェック方法**: 「Week 1 の合計工数は何時間ですか?キセ1人での作業を前提に現実的ですか?」と聞く。

#### 観点7: 認証情報の暗号化方式

仕様書では「暗号化保存」と書いているが、実装方式は決まっていない。AES-256? Fernet? 鍵管理は環境変数? KMS連携の余地は?

**チェック方法**: 暗号化の具体方式と鍵ローテーション方針を確認。

#### 観点8: 設計指針(design_guidelines.md)に準拠しているか

- ファイル分割が指針(1ファイル300行推奨)を踏まえた構造になっているか
- Week 1 のチケットに共通モジュール作成(logger / retry / encryption / repository基底等)が含まれているか
- AI Provider抽象化レイヤの実装が、Provider追加時のテストケース(セクション3.5)を満たすか
- マルチテナント分離が論理フィルタとRLSの二重チェック構成になっているか

**チェック方法**: implementation_plan.md の「設計指針との整合確認」セクションが書かれているか、そこで上記項目に触れているかを見る。書かれていない場合は追記を依頼。

### C.2 任意確認の観点

以下は要件未満でも次に進めるが、できれば確認したい点。

- ロギング設計(構造化ログ vs テキストログ、ログレベル運用)
- メトリクス/監視(Prometheus等の導入有無)
- テスト戦略(単体・統合・E2Eのカバレッジ目標)
- CI/CD(GitHub Actions の活用方針)
- ドキュメンテーション(API仕様書、ER図の自動生成)

## Part D: 実装開始後の運用フロー(参考)

`implementation_plan.md` の承認後、Week 1 から実装が始まります。各 Week の終わりに以下の流れで進めると、品質を保ちつつ後戻りを減らせます。

### D.1 Week 終了時のチェックポイント

- Claude Code が Week のタスクを完了したと報告
- キセが動作確認(仕様書セクション17の受け入れ基準のうち、その Week で達成すべき項目)
- 不具合・改善点をフィードバック
- 次の Week のタスクを再確認(必要なら調整)

### D.2 仕様変更が発生したとき

実装中に「やっぱりこの機能を足したい」「順序を変えたい」という変更が出ることは想定されます。その場合:

1. 仕様書(`ai_webmarketer_requirements_v2.1.md` → `v2.2.md` 等)を先に更新
2. 変更点を `implementation_plan.md` に反映
3. 影響範囲を Claude Code に評価させてから実装に着手

「仕様書を後追いで更新する」運用にしないこと。商材化を見据えるなら、仕様書が真実の源(Single Source of Truth)である状態を保つ。
