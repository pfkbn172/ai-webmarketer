## 1. プロダクト概要

### 1.1 何を作るか

中小企業向けにSEO/LLMO運用を自動化するSaaS型ウェブマーケティング支援システム。データ収集・AI分析・施策提案・効果測定を一気通貫でこなし、ダッシュボードで状況を可視化する。

### 1.2 想定ユーザー

| 段階 | ユーザー | 用途 |
|---|---|---|
| Phase 1(自社運用) | 株式会社kiseeeen(キセ) | 自社の集客実証 |
| Phase 2(クライアント展開) | はるき法律事務所など個別クライアント | 受託マーケ運用 |
| Phase 3(商材化) | 中小企業のWeb担当者・経営者 | 月額課金SaaS |

### 1.3 提供価値

1. **自動でデータが集まる**: GSC・GA4・主要LLM・競合情報を勝手に集約
2. **AIが助言する**: 月次・週次でAIが分析・施策提案
3. **AIが代行する**: コンテンツ・通知・SNS投稿を自動化
4. **見える化**: ダッシュボードで進捗・効果を一望
5. **複製可能**: 設定変更だけで別クライアント環境を立ち上げ
6. **AIモデル選択自由**: 用途ごとにGemini/Claude/GPT/Perplexityを切替可能

### 1.4 商材化の経済性目標

- 月額利用料: ¥38,000(計画v1から踏襲、要再検証)
- クライアントあたり月次運用コスト: Gemini無料枠活用で ¥500〜¥2,000
- 想定スケール: 10社で粗利率92%以上
- 複製1クライアントの立ち上げコスト: 1日以内の設定作業

## 2. 設計原則

### 2.1 AIを使う/使わないの境界

データ収集・整形・送信・スケジューリングはコード、判断・生成・自然言語処理はAIに任せる。

| 判断 | 担当 |
|---|---|
| データ収集・API呼び出し・整形 | コード |
| スケジューリング | コード |
| 異常検知(閾値比較) | コード |
| DB読み書き | コード |
| Webhook処理 | コード |
| **AI引用判定(回答中に自社が含まれるか)** | **コード**(文字列マッチ) |
| 月次レポートの分析・提言生成 | AI(デフォルトGemini、切替可) |
| コンテンツテーマ・タイトル候補生成 | AI(デフォルトGemini、切替可) |
| 記事ドラフト生成 | AI(デフォルトGemini、切替可) |
| 自然言語→構造化変換(問い合わせログ) | AI(デフォルトGemini、切替可) |
| 業種別規程チェック(弁護士広告等) | AI(デフォルトGemini、切替可) |
| LLMO観点のE-E-A-Tギャップ分析 | AI(デフォルトGemini、切替可) |

**重要**: 「AI処理」と「AI引用モニタリング対象LLM」は別概念である。前者は内部処理用に1つのLLMを呼ぶ、後者は5社のLLMに同じクエリを投げて引用状況を判定する。混同しないこと。

### 2.2 AI Provider 抽象化レイヤ(本バージョンの中核設計)

すべてのAI処理は、抽象化されたインターフェースを介して呼び出す。実装はAdapterパターン。

```
[AIユースケース層]
  monthly_report / content_draft / theme_suggestion / compliance_check / inquiry_structuring / eeat_analysis
                          ↓
[AI Provider Interface]   ← 統一インターフェース(generate, generate_structured, count_tokens等)
                          ↓
[Adapter実装]
  GeminiAdapter / ClaudeAdapter / OpenAIAdapter / PerplexityAdapter
```

- 各ユースケースは「どのProvider・どのModel・どのプロンプトテンプレートを使うか」を設定で指定
- Provider の APIキーはクライアントごと/または管理者共通で設定可能
- ユースケース×Provider×Modelの組合せをDBに保持
- 切替時はコード変更不要、設定変更のみで反映

#### 2.2.1 デフォルト設定(Phase 1)

| ユースケース | デフォルトProvider | デフォルトモデル | 理由 |
|---|---|---|---|
| 月次レポート生成 | Gemini | gemini-2.5-flash | 大量トークン処理、無料枠が広い |
| コンテンツテーマ提案 | Gemini | gemini-2.5-flash | 同上 |
| 記事ドラフト生成 | Gemini | gemini-2.5-pro | 文章品質要求が高い、Proを使う |
| 規程チェック | Gemini | gemini-2.5-flash | ルールベース判定が中心 |
| 問い合わせ構造化 | Gemini | gemini-2.5-flash | 短い処理、Flashで十分 |
| E-E-A-Tギャップ分析 | Gemini | gemini-2.5-flash | 数値・構造分析中心 |

#### 2.2.2 切替例(Phase 2以降の運用)

- 「文章品質を上げたい記事」だけClaudeに切替: content_draft = Claude / claude-sonnet-4-7
- 「リアルタイムWeb情報が必要なテーマ提案」をPerplexityに: theme_suggestion = Perplexity / sonar-large
- 「コーディング系記事」だけClaudeに: 設定でテンプレート別に分岐

### 2.3 マルチテナント設計

- 1つのシステムインスタンスが複数クライアントを管理する
- データはクライアントID(tenant_id)で論理分離
- 設定値はクライアントごとに分離(認証情報、ターゲットクエリ、著者プロフィール、Providerモデル設定)
- ダッシュボードはクライアント切り替え可能

### 2.4 既存VPS環境の活用

- 本システムは既存VPS上の `app.kiseeeen.co.jp/marketer/` パスベースで配置
- Nginxリバースプロキシで `/marketer/` 配下へルーティング
- 静的アセット・SPA・APIすべてサブパス対応必須
- 既存のClaude Code開発フロー(GitHub→git pull→デプロイ)を活用
- バックアップは既存のpg_dump + rclone + Backblaze B2の構成を流用
- ConoHa VPSのディスクI/O障害の経験を踏まえ、データの定期バックアップ・複数箇所保管は必須

### 2.5 LLMO最優先

LLMO関連機能は省略不可の核心要素。データ収集層・分析層・ダッシュボード層のすべてに統合する。SEOは旧世代の検索順位を見るため、LLMOはAI検索時代の引用率を見るため。両方必要だが、商材としての差別化はLLMO側にある。

## 4. 機能要件

### 4.1 データ収集機能

#### 4.1.1 Google Search Console連携

- 認証: OAuth2(クライアントごとに認証情報を保持)
- 取得データ: クエリ別クリック・表示回数・CTR・平均掲載順位、URL別クリック・表示回数・順位
- 取得頻度: 週次(月曜早朝)
- 保存先: PostgreSQL

#### 4.1.2 Google Analytics 4連携

- 認証: OAuth2
- 取得データ: セッション・ユーザー・直帰率・コンバージョン・流入元・ページ別パフォーマンス
- 取得頻度: 週次
- 保存先: PostgreSQL

#### 4.1.3 AI引用モニタリング(LLMO核心機能)

主要LLMにターゲットクエリを定期的に問い合わせ、自社/クライアント名が回答に含まれるかを記録する。

**重要**: ここで呼ぶLLMは「2.2 AI Provider 抽象化レイヤ」とは別概念。引用モニタはハードコードで5社全部を呼ぶ。

| 対象LLM | アクセス方法 | 備考 |
|---|---|---|
| ChatGPT(OpenAI) | OpenAI API(Web検索付きモデル) | gpt-4oで検索ON |
| Claude(Anthropic) | Anthropic API(Web検索ツール付き) | Sonnet 4でweb_search使用 |
| Perplexity | Perplexity API | API利用 |
| Gemini | Google AI Studio API | Web grounding付き |
| Google AI Overviews | SerpApi or DataForSEO | スクレイピングAPI経由 |

- 取得頻度: 週次(クエリ数 × LLM数 × 週1回)
- 取得項目: クエリ、LLM名、回答全文、引用URL一覧、自社言及有無、競合言及有無、取得日時
- 引用判定ロジック: 回答テキストに以下のいずれかが含まれるか文字列マッチ
  - クライアントのドメイン名(例: kiseeeen.co.jp)
  - クライアントのブランド名(例: kiseeeen, 株式会社kiseeeen)
  - クライアントの代表者名(例: 黄瀬剛志)
  - 引用URL一覧に自社ドメインが含まれるか
- 競合の引用も同時にカウント(設定で競合リストを定義)
- ターゲットクエリ数の目安: 1クライアントあたり20〜30本

#### 4.1.4 構造化データ監査(LLMO支援機能)

- 対象: クライアントサイトの主要URL(設定で指定)
- 取得方法: 各URLにHTTP GETしてHTMLを取得、JSON-LDブロックを抽出・パース
- チェック項目:
  - Article schema の存在と必須フィールドの充足
  - FAQPage schema の存在(該当ページのみ)
  - Organization schema(サイト全体)
  - Person schema(著者ページ)の存在と完全性
  - BreadcrumbList の有無
- 出力: 充足度スコア(各記事ごと0〜100)+ 不足フィールドのリスト
- 取得頻度: 月次

#### 4.1.5 競合RSS / コンテンツ収集

- 設定で指定した競合サイトのRSSフィードを取得
- 競合の最新記事タイトル・URL・公開日を記録
- 取得頻度: 週次

### 4.2 LLMO支援機能

#### 4.2.1 ターゲットクエリ管理

- 1クライアントあたり最大100本のクエリを登録可能
- クエリにメタ情報を付与: クラスタID(Pillar/Cluster構造を表現)、優先度、想定検索意図、転化期待度
- ダッシュボードから追加・削除・編集可能
- クエリごとの「現状LLM引用状況」と「目標(月間引用回数)」を表示
- CSV一括インポート/エクスポート対応

#### 4.2.2 著者プロフィール管理(E-E-A-Tシグナル)

各クライアントに対し、著者プロフィールを構造化して保持する。これは構造化データ自動付与とAI助言生成の両方で使う。

著者プロフィール項目(JSON):
- name: 表示名
- jobTitle: 役職
- worksFor: 所属組織
- alumniOf: 学歴・出身組織
- credentials: 保有資格(配列)
- expertise: 専門領域(配列)
- publications: 著作・出版物(配列、URL付き)
- speakingEngagements: 登壇・取材実績(配列)
- awards: 認定・受賞(配列)
- bioShort: 短い自己紹介(150字以内)
- bioLong: 詳細経歴(1000字以内)
- socialProfiles: SNS等の外部プロフィールURL

#### 4.2.3 llms.txt 自動生成

- ルートに置く llms.txt をシステムが自動生成
- 内容: サイト概要、主要記事URL一覧、著者情報、ライセンス
- 記事公開時に自動更新
- 更新後は対象サイトに自動デプロイ(WordPress REST API or Webhook経由)

#### 4.2.4 構造化データ自動付与

- 新規記事公開時、Webhookで通知を受けてArticle/FAQPage/Person スキーマを自動生成
- 著者プロフィールから Person スキーマを引っ張る
- 生成したJSON-LDをWordPressに書き戻す(REST API)
- 既存記事に対しても一括適用機能を提供

#### 4.2.5 引用機会分析

- 「競合が引用されているがクライアントが引用されていないクエリ」を抽出
- 各クエリに対し、引用獲得のための推奨アクション(著者プロフィール強化、特定構造化データ追加、コンテンツ補強等)をAI(デフォルトGemini)が提案
- ダッシュボードで一覧表示、クリックで詳細

### 4.3 AI分析エンジン

すべて 2.2 のAI Provider抽象化レイヤを通じて呼ぶ。デフォルトはGemini、設定で切替可能。

#### 4.3.1 月次レポート生成

- 毎月1日に自動実行
- 入力: 過去30日の全データ(GSC・GA4・AI引用ログ・コンテンツ・問い合わせ・競合)
- 出力: HTMLレポート(構造化された分析+提言)
- レポート構成:
  - サマリ(全体KPIの推移)
  - SEO観点の分析
  - **LLMO観点の分析**(AI引用率推移、引用機会、E-E-A-Tギャップ)
  - コンテンツ施策の効果検証
  - 来月の推奨アクション(優先度付き5〜10項目)
  - 次月のコンテンツテーマ提案(5〜10本)
- 配信: Resend(メール)+ ダッシュボードで履歴閲覧

#### 4.3.2 週次サマリ

- 毎週月曜にAI生成の短いサマリ
- 内容: 今週のハイライト・異常検知・即時アクション推奨
- 配信: メール+Slack/Teams(設定で選択)

#### 4.3.3 コンテンツテーマ提案

- 引用機会分析の結果と、著者プロフィールの「書ける領域」を組み合わせて提案
- 各テーマに想定タイトル・想定見出し構成を付与
- ダッシュボードから「採用」をクリックすると次の機能(記事ドラフト生成)に渡る

#### 4.3.4 記事ドラフト生成

- 採用したテーマに対し、AIでMarkdownドラフトを生成
- プロンプトには以下を組み込む:
  - 著者プロフィール(完全情報)
  - クライアントの独自フレームワーク・専門用語(設定)
  - 業種別の規程・禁止表現(設定、例: 弁護士広告規程)
  - LLMO最適化フォーマット(冒頭定義文、表、FAQ、E-E-A-Tシグナル)
- 生成後はDBに保存、ダッシュボードでレビュー
- レビューOK後、WordPress下書き投稿

#### 4.3.5 業種別規程チェック

- 弁護士広告規程など、業種固有のチェックを行う
- 各クライアントの業種設定に応じて適用ルールを切り替え
- 記事ドラフトに対し、規程違反の可能性がある表現を指摘

### 4.4 Webhook / トリガー機能

#### 4.4.1 ブログ公開トリガー

- 入力: WordPress(または他のCMS)からのWebhook POST
- 処理:
  1. 通知メール送信(Resend)
  2. SNS自動投稿(設定で有効な場合のみ。MVPでは無効デフォルト)
  3. DBのコンテンツ台帳に記録
  4. **構造化データ自動付与**(Article + FAQPage + Person)
  5. **llms.txt 更新**
  6. AI引用モニタの対象URL更新
  7. MEO投稿文ドラフト生成(AI)→ DBに保存

#### 4.4.2 問い合わせ受信トリガー

- 入力: 問い合わせフォームのWebhook、または音声テキストの手動投入
- 処理:
  1. AIで構造化(日付・業種・規模・内容・経路・ステータス・**AI起点かどうか**)
  2. DBの問い合わせログに保存
  3. ダッシュボードに即時反映

#### 4.4.3 異常検知トリガー

- 順位急落、AI引用率の急減、サイト到達不能、構造化データ破損などを検知
- 検知時にメール通知

### 4.5 Webダッシュボード

#### 4.5.1 ログイン・認証

- メール+パスワード認証(MVP)
- マルチテナント前提でユーザーごとに見えるクライアントを制御
- 管理者ロール(全クライアント)・クライアントロール(自分のクライアントのみ)を分離
- Phase 2でSSO・OAuth対応を検討

#### 4.5.2 ダッシュボード(KPI俯瞰画面)

トップ画面。1クライアント分のKPIを一画面で把握できる。

表示要素:
- 期間選択(過去7日/30日/90日)
- KPIカード群(横並び):
  - **AI引用回数**(今月実績、前月比)
  - **AI引用率**(今月実績、前月比)
  - **指名検索数**(GSCより)
  - **オーガニックセッション**(GA4より)
  - **問い合わせ数**(うちAI起点を別表示)
  - **公開記事数**(今月)
- 推移グラフ(折れ線):
  - AI引用回数の日次推移
  - 検索順位推移(主要KW 5本)
  - セッション推移
- アラート/推奨アクション(上位3〜5件、AI生成)
- 直近の活動ログ(コンテンツ公開、レポート生成、問い合わせ受信)

#### 4.5.3 AI引用モニタ画面

- クエリ別の引用状況テーブル
  - 列: クエリ、ChatGPT、Claude、Perplexity、Gemini、AI Overviews
  - セルの値: 直近4週の引用カウント

## 5. データモデル(概念設計)

詳細なテーブル定義はClaude Codeに委ねる。以下は概念モデルとリレーション。

### 5.1 主要エンティティ

```
Tenant (クライアント)
  ├─ id, name, industry, domain, compliance_type, created_at, ...
  ├─ Settings (1:1)
  ├─ AuthorProfile (1:1)
  ├─ Users (1:N)
  ├─ TargetQueries (1:N)
  ├─ Competitors (1:N)
  ├─ Contents (1:N)
  ├─ Inquiries (1:N)
  ├─ Reports (1:N)
  ├─ Credentials (1:1, encrypted)
  └─ AIProviderConfigs (1:N) ← 用途×Provider×Modelの組合せ

AIProviderConfig (AI Provider設定)
  ├─ id, tenant_id
  ├─ use_case (enum: monthly_report, content_draft, theme_suggestion, ...)
  ├─ provider (enum: gemini, claude, openai, perplexity)
  ├─ model_name (string)
  ├─ prompt_template_id (FK)
  ├─ temperature, max_tokens等のパラメータ
  └─ is_active

PromptTemplate (プロンプトテンプレート)
  ├─ id, name, use_case, version
  ├─ template_text (placeholders含む)
  └─ created_at, updated_at

TargetQuery (ターゲットクエリ)
  ├─ id, tenant_id, query_text, cluster_id, priority, expected_conversion
  └─ CitationLogs (1:N)

CitationLog (AI引用ログ)
  ├─ id, tenant_id, query_id, llm_provider, query_date
  ├─ response_text, cited_urls (JSON)
  ├─ self_cited (bool), competitor_cited (JSON)
  └─ created_at

Content (コンテンツ)
  ├─ id, tenant_id, url, title, status, published_at
  ├─ pillar_id, cluster_id (記事のトピッククラスタ位置)
  ├─ schema_score (構造化データ充足度)
  └─ Metrics (1:N、日次のセッション・引用数等)

Inquiry (問い合わせ)
  ├─ id, tenant_id, received_at
  ├─ industry, size, content_text
  ├─ source_channel (web/email/phone/ai)
  ├─ ai_origin (chatgpt/claude/perplexity/gemini/aio/null)
  └─ status

Report (月次レポート)
  ├─ id, tenant_id, period (yyyy-mm)
  ├─ summary_html, action_plan_json
  └─ generated_at

KPILog (日次KPIスナップショット)
  ├─ id, tenant_id, date
  ├─ sessions, clicks, impressions, avg_position
  ├─ ai_citation_count, named_search_count
  └─ inquiries_count

AuthorProfile (E-E-A-Tシグナル)
  └─ tenant_id, name, jobTitle, credentials (JSON), publications (JSON), ...
```

### 5.2 マルチテナント分離方針

- すべての主要テーブルに tenant_id を持たせる
- アプリケーション層で必ず tenant_id によるフィルタを掛ける
- 将来的にDBスキーマ分離に切り替え可能な設計にしておく
- Row Level Security(RLS)の活用も検討

### 5.3 認証情報の取り扱い

- API認証情報(OAuth、APIキー)は暗号化して保存
- 暗号化キーは環境変数で管理し、コードリポジトリに含めない
- AI ProviderのAPIキーは「テナント別」「システム共通」の両方を許容(設定で選択)

## 8. エラーハンドリングと信頼性

### 8.1 リトライ

- 外部API呼び出しは指数バックオフで最大3回リトライ
- 失敗時はエラーログをDBに記録、管理者に通知

### 8.2 フォールバック

- AI Provider呼び出し失敗時、設定で「フォールバック先」を指定可能(例: Gemini失敗→Claudeで再実行)
- LLMの一部が呼び出し失敗しても他のLLMは続行

### 8.3 バックアップ

- DBは日次でpg_dump、rclone経由でBackblaze B2にアップロード(既存運用を流用)
- バックアップ世代は7日間+月次1年間

### 8.4 監視

- ジョブ実行成功/失敗のメトリクスをUptimeRobotまたは類似サービスで監視(既存運用を流用)
- 異常時はメール通知

## 9. セキュリティ

### 9.1 認証情報管理

- すべてのAPIキー・OAuth tokenは暗号化してDB保存
- 環境変数経由で暗号化キーを供給
- 平文をログ・コードリポジトリに含めない

- ダッシュボードはBASIC認証(MVP)→ アプリケーション認証(Phase 1完了時)
- HTTPS必須(Let's Encryptは既存VPSで稼働中)
- IPアドレス制限(クライアント側設定でON/OFF可能)
- `app.kiseeeen.co.jp/marketer/` 配下のすべてのエンドポイントで認証必須

### 9.3 データ取り扱い

- 問い合わせログには個人情報が含まれる可能性があるため、表示時は氏名・連絡先をマスク表示(設定で切替)
- DB暗号化(at rest)はPostgreSQLの暗号化機能で対応

## 11. ホスティングとデプロイ

### 11.2 想定構成

│   │   ├─ 静的アセット(SPA build)

- GitHubリポジトリ → VPSで `git pull` → `docker compose up -d --build` の流れ(既存と同じ)

## 14. 技術選定の指針(Claude Code判断)

技術スタックの選択は実装担当(Claude Code)に委ねる。ただし以下の方針を満たすこと。

### 14.1 推奨

- バックエンド: Python(FastAPI想定だがClaude Code判断)
- DB: **PostgreSQL一本**(暗号化・JSON型・RLSサポートが必要)
- フロントエンド: React + TypeScript(または同等のSPA、Claude Code判断)
- スタイリング: TailwindCSS推奨(一貫したダークUI、編集者デザインに寄せやすい)
- 認証: 自前実装(MVP)→ Auth0等の検討はPhase 2
- スケジューラ: cronまたはGHA scheduleの併用
- ホスティング: 既存VPS上にDocker Compose構成

### 14.2 制約

- 既存の `app.kiseeeen.co.jp` 配下に `/marketer/` パスベースで同居できる構成にすること
- 既存のClaude Code開発フロー(GitHub→git pull→デプロイ)を活用できること
- 過去のVPS障害(ConoHa jbd2/vda2 hang)を踏まえ、データ永続性に妥協しないこと
- ライブラリのライセンスは商用利用可能であること
- AI Provider抽象化レイヤを作り、Gemini以外のProviderを追加するときコード変更を最小化すること

### 14.3 デザイン方針

- ダッシュボードのデザインは編集的・落ち着いた色調を選ぶ(派手なSaaS感は避ける)
- 中小企業の経営者・Web担当者が見て即理解できるシンプルな情報設計
- ダッシュボードのKPIカードは数字を大きく、推移を小さく
- グラフはRecharts等のシンプルなチャートライブラリ
- ダークUIをデフォルトとし、明るいテーマも切替可能

## 16. リスクと対応策

| リスク | 影響 | 対応 |
|---|---|---|
| LLMのAPI仕様変更 | 引用モニタ停止 | 各LLMの抽象化レイヤを設け、変更時は1箇所で対応 |
| Google AI Overviews取得手段の不安定さ | LLMO分析の一部欠損 | SerpApi・DataForSEO・自前スクレイピングの3系統を持つ |
| クライアント認証情報の漏洩 | 信用失墜 | 暗号化保存、最小権限、アクセスログ監査 |
| VPS障害(過去に発生) | サービス全停止 | バックアップ厳格化、Phase 3で別リージョンへの自動フェイルオーバ検討 |
| Geminiの無料枠変更 | コスト増 | AI Provider切替が前提なので別Providerに即移行可能 |
| Gemini AI処理品質が要求に届かない | レポート品質低下 | 用途別にClaude/GPTに切替、設定で制御 |
| 競合の参入(同種SaaSの登場) | 商材化失敗 | キセの独自性(タイ製造業経営層経験、独自フレーム)で差別化、汎用ツールではなく業種特化を売る |
